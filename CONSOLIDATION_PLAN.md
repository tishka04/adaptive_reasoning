# Plan de consolidation — vers une architecture « théorie du jeu »

> But : arrêter la dispersion entre `v3/`, `v4/`, `v4_1_reasoning_system/`, `v5/`,
> et converger vers **une seule boucle** :
> `exploration → hypothèses de mécanique → expérience discriminante → confirmation/réfutation → révision du Task Program → plan`.
>
> Constat de fond (confirmé par le code) : la couche « hypothèse » actuelle est
> *locale par action* et re-convertie en score — c'est un habillage de `état → ML → action`.
> Le raisonnement causal relationnel (prédicats sur objets, testés délibérément, révisés) **n'existe nulle part**.

---

## 1. Audit des quatre architectures

### v3/ — substrat symbolique (le plus proche de la cible)
- `v3/mechanics/operator_inducer.py` — induit des opérateurs (move/noop/lethal/click/global_transform) avec support/contradictions.
- `v3/mechanics/rule_engine.py` — **règles causales relationnelles** : `AdjacentToValue`, `ObjectCount`, `RemovesObject`, `GameOverEffect`… avec support/contradictions et vérification. **C'est la pièce relationnelle la plus avancée du dépôt.**
- `v3/mechanics/experiment_designer.py` — sélection d'expérience par gain d'information, mais **sur le *type d'effet* d'une action isolée** (entropie sur movement/noop/lethal…), pas sur la discrimination entre règles concurrentes.
- `v3/memory/belief_debugger.py` — **réfutation/dégradation** de croyances contredites. Réutilisable tel quel.
- `v3/control/` — specialist_minds, arbiter, operator_search, options, progress_tracker.

### v4/ — compétition d'ontologies + zoo de modèles
- `v4/ontology/ontology_competition.py` — **maintient plusieurs théories du jeu en compétition** (avatar/click/token/field/phase/transform world) mises à jour par evidence_for/against. C'est le bon *mécanisme* (théories concurrentes révisées), mais une **taxonomie figée à 6 classes**, et la sortie n'est qu'un biais d'affordance (→ scorer).
- `v4/learning/` — 14 petits modèles (value, sterility, reliability…). Périphérie.

### v4_1_reasoning_system/ — pile neuronale « predict then act »
- `reasoning_loop.py` (115 Ko), `visual_cortex.py`, `game_world_model.py` (JEPA), `energy_scorer.py` (EBM).
- `hypothesis_models.py` / `dynamics_inducer.py` / `trajectory_sampler.py` — hypothèses **locales par action** re-scorées (`_hypothesis_score`).
- `latent_task_programmer.py` — dérive un « task program » **depuis le scorer lui-même → circulaire**.
- Task programs (`task_programs/*.json`) : efficaces mais **compilés depuis trace humaine**, pas auto-générés/révisés.

### v5/ — v3 étendu
- Copie de v3 + `danger_memory`, `anti_attractor`, `learned_priors`, `bissociation_lite`, `dissent_controller`, `ritual_store`, `win_digest`.

## 3. Architecture cible (une seule boucle)

```
            ┌────────────────────────────────────────────────┐
            │            BELIEF / THEORY STORE                │
            │  GameTheory = { MechanicHypothesis* }           │
            │   MechanicHypothesis : prédicat relationnel     │
            │     (objets, correspondances, contraintes)      │
            │     + support / contradictions / statut         │
            └───────┬───────────────────────────▲────────────┘
                    │ théories concurrentes      │ révision (belief_debugger)
                    ▼                            │
   exploration ─▶ ExperimentDesigner ─▶ action ─▶ observation ─▶ refute/confirm
   (bootstrap)    « quelle action SÉPARE          (env)          (rule_engine.verify)
                   le mieux H_i de H_j ? »                              │
                    │                                                   ▼
                    └───────────── plan multi-pas conditionné à la théorie dominante
```

Différence clé vs `experiment_designer` actuel : l'expérience maximise la **divergence de prédiction entre théories concurrentes** (« si H1, le clic déplace la forme ; si H2, il la supprime »), pas le gain d'information sur une action isolée.

## 4. Premier incrément (après ce plan)

1. `MechanicHypothesis` relationnel + `GameTheory` (conteneur de théories concurrentes), bâtis sur `v3/schemas.py`.
2. `DiscriminatingExperimentDesigner` : sélectionne l'action maximisant la divergence de prédiction entre les 2 théories de tête.
3. Boucle de révision : `rule_engine.verify` + `belief_debugger` → confirme/réfute → met à jour `GameTheory`.
4. **Métrique** (option 3, à faire en parallèle) : sur `ar25` avec trace humaine de référence — nombre d'hypothèses correctement confirmées/réfutées, et pas d'écraser le score local.

Cible du prototype : prouver la boucle hypothèse→expérience→révision sur `ar25` (jeu instrumenté), **pas** maximiser un score.
